一句话区分 deadlock / data race / race condition
标准答案：
deadlock（死锁）：多个线程互相等待资源（锁），谁也继续不了。
data race（数据竞争）：并发访问同一内存位置，至少一个写，且缺少同步 → 未定义行为。
race condition（竞态条件）：程序结果依赖执行时序（逻辑层面的），不一定是未定义行为，但可能出现错误结果。
一句话版：
死锁是卡住；data race 是无同步读写同变量导致 UB；race condition 是时序导致逻辑错。
6) Q6-追问A：给两个“不能丢 vs 可丢旧保新”的消息例子

你做机器人/中间件很贴切，直接用这两个：
不能丢（要可靠）例子：
控制指令/开关量：例如“紧急停止 E-stop”“刹车/上电/解锁”“模式切换”。
理由：丢一次可能直接安全事故或状态错误。
可丢旧保新（保最新）例子：
状态/遥测/心跳：例如“电池电量/温度/定位/速度状态”“心跳包”。
理由：状态是不断刷新，旧的很快过期，保最新更合理；高负载时 drop-old 能防止堆积和延迟扩散。
一句话版：
命令类不丢，状态类可丢旧保新。

) MLen / Header 的端序怎么判断？（你自己就能验证）

真实协议当然重要，但你现在学习也可以先假设；不过学会“如何判断端序”本身就是系统岗面试加分项。

方法A：用一个“已知长度”的包反推（最靠谱）

你随便抓到一帧数据（抓包、日志、dump 都行），只要能看到：

header 后面紧跟的 2 字节（len）

以及后面 body 的实际字节数

然后对比：

如果 len 两字节是 00 2A，body 实际长度是 42 → 大端

如果 len 两字节是 2A 00，body 实际长度是 42 → 小端

你哪怕不知道 header，只要找一个你确定 body 是 JSON 的地方（例如看到 {），再数 body 到结尾的字节数，就能反推 len。

方法B：在代码里打印 len 两种解释（最快）

接收端拿到 len 的两个字节 b0 b1，同时计算：

be = (b0<<8)|b1

le = (b1<<8)|b0

然后看哪一个更像“合理长度”（比如几十~几千字节）。另一个往往会变成很离谱的数（比如 12000+）。

方法C：判断 header 在字节流的排列

你说 FHD = 0xAAAA5555。如果网络里看到的是：

AA AA 55 55 → 大概率按大端发送

55 55 AA AA → 大概率按小端发送（或发送端直接把 uint32_t 内存拷过去）

这也是一个很好的工程判断题。

**学习建议：**你不需要现在就知道答案，但要知道“我怎么确认”。面试时你把上面 A/B 说出来，非常加分。

4) 你的学习路线怎么优化？（比“LeetCode+ROS2+八股文”更聚焦）

你之前那套风险是：样样学但不形成作品。系统岗跳槽要的是“可讲、可演示、可复盘”。

我建议你改成“三条主线 + 一条保底”：

主线A：C++工程化（每天30–40分钟）

目标：能写出异常安全、低拷贝、可维护的代码
内容：RAII、Rule of 5、STL容器、string/byte、错误处理、单元测试

主线B：并发与消息架构（每天30–40分钟）

目标：能把“回调-队列-worker-背压”讲清楚并写出来
内容：mutex/cv、bounded queue、线程池、丢弃策略、延迟与吞吐

主线C：协议/中间件（每周2次，每次60分钟）

目标：能做一个“可靠拆包 + 分发 + 重连/超时”的小系统
内容：拆包器、状态机、超时重试、灰度兼容（JSON→Protobuf）

保底：LeetCode（每天15–20分钟）

只练系统岗常见题型，目的是不掉链子，不追高难。

ROS2：先放到第2个月再系统补。你先把“并发+协议+工程化”打穿，立刻就能变成简历卖点。

## 错位竞争
你“错位竞争抓机器人方向”是否正确？
我认为是正确的，而且你有三个天然优势：
✅ 优势1：你不是“纯写业务代码”，你做的是系统软件
中间件、协议、线程模型、数据流，这些是机器人团队很缺的人。
✅ 优势2：你能把“机器人场景约束”讲清楚
比如：
实时性/延迟抖动
传感器数据吞吐
控制闭环可靠性
现场排障
这不是纯互联网/纯嵌入式的人天然具备的表达框架。
✅ 优势3：机械本科不是劣势，反而可包装为“跨学科”
前提是你要把叙事改成：
我不是靠机械知识吃饭，但我能理解机器人系统从物理约束到软件架构的全链路，这让我在系统设计和联调上更高效。
你担心“打不过 985/211 纯科班”——确实在纯算法/纯嵌入式底层某些方向会吃亏，但你选的赛道（机器人系统软件/架构偏）本来就是工程经验+系统思维更值钱。
