CyberRT → Foxglove Bridge 系统设计报告
1. 系统背景与设计动机

随着机器人系统逐步从传统 ROS 生态向自研中间件体系演进，当前系统已从 ROS1/ROS2 架构逐步过渡到以 CyberRT 为核心的数据通信框架。在这一过程中，原有依赖 RViz 的可视化调试体系逐渐暴露出强耦合、难以远程调试以及不适配多中间件系统的问题。

RViz 本质上依赖 ROS 的 TF、topic graph 以及 message type support，其能力与 ROS 强绑定，导致在去 ROS 化的系统中无法直接复用。同时，在实际工程中，远程调试、多源数据融合以及跨系统可视化已成为核心需求，这使得传统本地 GUI 工具逐渐无法满足工程化要求。


基于上述背景，本系统设计并实现一个轻量级 Bridge 中间层，用于将 CyberRT 中的实时数据流转换为 Foxglove Studio 可识别的标准可视化数据协议，从而实现 ROS/RViz 体系的替代，并构建统一的跨平台调试通道。

2. 系统目标与设计原则

本 Bridge 系统的核心目标是构建一个低延迟、高扩展性的数据转换与转发层，使 CyberRT 内部的传感器与状态数据能够无损地映射至 Foxglove 可视化前端。系统不承担任何算法职责，也不参与数据融合或决策逻辑，而仅作为数据语义转换与传输通道存在。

在设计原则上，系统强调轻量化与解耦能力。所有上游数据应保持 CyberRT 原生语义，而所有下游输出则严格遵循 Foxglove WebSocket 协议规范。系统必须支持高频数据流的实时转发，同时保证在极端负载情况下具备可控的延迟与丢包策略。

此外，系统需要具备良好的扩展能力，以支持未来新增传感器类型、topic 数量扩展以及多实例 Bridge 并行运行的可能性。

3. 系统总体架构设计

本系统采用分层式架构设计，将整个数据流处理过程划分为 CyberRT 接入层、语义统一层、核心调度层以及 WebSocket 输出层四个主要部分。

在最上层，CyberRT 接入层负责与底层通信框架进行对接，通过订阅机制获取各类 sensor topic 数据。该层仅负责数据接收与初步解析，不参与任何跨 topic 语义处理。

在接收层之下，系统引入统一消息抽象层，用于将不同来源的数据结构统一转换为标准内部格式。该层的设计是整个 Bridge 系统的关键，其核心作用在于消除 CyberRT 与 Foxglove 在数据模型上的差异，并引入统一的时间戳、frame_id 以及 topic 元信息，从而为后续调度与可视化提供一致的数据语义基础。

在核心调度层中，系统实现了基于 topic 的路由与多线程调度机制。该层负责根据 topic 类型进行分类处理，并将数据分发至不同的处理队列中。为了满足低延迟要求，系统采用基于队列缓冲与线程池结合的方式进行并发处理，同时引入背压控制机制，避免在高频数据场景下出现系统阻塞。

最下层为 WebSocket 输出层，该层将统一后的数据编码为 Foxglove 协议所定义的 Protobuf 消息格式，并通过 WebSocket 长连接发送至 Foxglove Studio。该层同时负责 message batching 与发送节流，以保证网络带宽利用效率与前端渲染稳定性。

4. 核心模块设计

系统整体由四个核心模块组成，分别对应数据接入、语义转换、调度处理以及输出传输四个关键阶段。

CyberRT 接入模块负责与 CyberRT runtime 进行集成，通过订阅机制获取实时 topic 数据流。该模块的设计目标是尽可能保持与 CyberRT 原生机制一致，从而减少额外的中间开销。所有接收到的数据将在该层完成初步 protobuf 解析，并交由上层进行统一处理。

在此基础之上，统一消息抽象模块承担整个系统的语义标准化职责。该模块将不同来源的数据封装为统一结构体，其中必须包含时间戳、frame 标识以及 payload 内容。该设计的关键意义在于屏蔽底层数据源差异，使上层调度逻辑与下层可视化协议解耦。

核心调度模块是整个 Bridge 的运行中枢，其负责对不同 topic 数据进行路由分发与线程调度。在该模块中，系统采用基于 topic 粒度的处理模型，将高频与低频数据分离处理，并通过队列缓冲机制实现流量平滑。此外，该模块还承担 QoS 控制功能，包括丢包策略、更新频率限制以及缓存深度管理。

最后，WebSocket 输出模块负责将处理后的数据转换为 Foxglove 可识别的标准协议，并通过长连接方式推送至前端。该模块不仅负责数据发送，还需要维护 channel 映射关系，并确保数据在传输过程中的顺序一致性与时间一致性。

5. 多线程与低延迟设计

由于系统需要同时处理高频传感器数据与低延迟可视化输出，因此整体架构采用多线程并发模型，但线程划分遵循功能解耦原则，而非简单的全局线程池模型。

CyberRT 数据接收通常运行在独立 IO 线程中，以避免回调阻塞。数据在进入系统后，将被快速分发至解析线程进行格式转换，随后进入 topic 级别的处理队列。不同 topic 可根据负载情况分配独立 worker thread，从而避免高频 topic 对低频 topic 的干扰。

在输出侧，系统单独设置 WebSocket 发送线程，用于统一管理网络 IO 操作，并通过批量发送策略降低系统调用开销。这种设计能够在保证并发性的同时，维持整体数据流的稳定性。

6. 软硬件与运行环境要求

本系统依赖 CyberRT 运行时环境以及 Protobuf 序列化库，同时需要支持标准 WebSocket 通信协议的网络库实现。在编译层面，系统采用 C++17 标准，以保证线程库与内存模型的稳定性。

在运行环境方面，系统建议部署于 x86 或 ARM64 架构设备，并至少具备四核 CPU 与千兆网络环境，以满足多传感器高频数据转发需求。在高负载场景下，建议使用八核及以上配置以保证稳定运行。

7. 系统风险与约束

尽管 Bridge 系统在架构上实现了 CyberRT 与 Foxglove 的解耦，但其本质仍然是一个语义转换层，而非完整机器人中间件。因此，系统存在若干不可忽略的结构性风险。

其中最核心的风险在于时间语义与坐标语义的缺失。Foxglove 本身并不提供类似 ROS TF 的缓存与插值机制，这意味着所有坐标一致性问题必须由上游系统或 Bridge 自行补偿。一旦时间戳不一致或 frame 构建不完整，将直接导致可视化结果错位。

此外，在高频数据场景下，系统可能面临队列积压与网络带宽瓶颈问题，进而导致数据延迟增加甚至丢包。因此在设计上必须引入合理的丢弃策略与流控机制。

最后，由于系统将原本属于 ROS 生态中的语义能力上移至业务层，整体系统复杂度将显著增加，这对算法与软件协作提出更高要求。

8. 总结

本 Bridge 系统本质上是一个面向去 ROS 化架构的实时数据语义转换层，其核心价值在于将 CyberRT 内部数据流无损映射至 Foxglove 可视化体系，从而实现跨系统调试与远程可视化能力。

然而，该系统并不等价于 RViz 替代方案的完全实现，而是将原本由 ROS 提供的语义能力拆解并上移至应用层。因此，在实际落地过程中，必须充分评估时间系统、TF 语义以及多源同步带来的工程复杂度，以避免系统性风险积累。
